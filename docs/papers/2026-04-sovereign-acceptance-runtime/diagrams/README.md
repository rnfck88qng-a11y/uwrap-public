# Diagram Suggestions

These diagrams should reinforce doctrine and bounded proof, not imply a finished platform.

## 1. Execution Boundary vs Model Layer

Show a generic model on one side and the Sovereign Acceptance Runtime as the bounded execution wall between model output and canonical state.

Purpose:
make clear that governance lives at the execution boundary, not inside model rhetoric.

## 2. Canonical Promotion Circuit

Flow:
proposal -> bounded execution -> receipts/evidence -> human review -> CANON

Purpose:
show the exact point where machine proposal ends and human acceptance begins.

## 3. Receipt-Driven Governance Flow

Chain:
instruction receipt -> execution receipt -> review receipt -> acceptance receipt

Purpose:
show governance as a typed receipt circuit, not just logging.

## 4. Tri-State Truth Matrix

Grid with rows as major claims and columns:
Specified / Partially Enforced / Proven

Purpose:
signal honesty and bounded proof immediately.

## 5. Projection vs Truth

Bottom layer:
ledger / canonical store

Top layer:
dashboard / UI / projection surface

Purpose:
show that UI is read-only projection, not source of truth.

## 6. Bounded Proof vs Open Problems

Two columns:

Left:
what is proven in bounded scope

Right:
what remains partial, specified, or contradictory

Purpose:
make the paper’s humility legible at a glance.
