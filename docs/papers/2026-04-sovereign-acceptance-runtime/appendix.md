# Technical Appendix
## The Sovereign Acceptance Runtime

This appendix exists to prevent doctrinal drift between the paper and the actual system state.

## 1. Truth classes used in the paper

Every major public claim should be constrained to one of these classes:

- **Proven** — multi-source and bounded by accepted evidence
- **Partially Enforced** — mechanism exists in bounded paths but is not yet generalized
- **Specified** — defined in doctrine, schema, or blueprint, but not operating as a proven path
- **Contradicted** — internally unresolved or materially inconsistent across sources
- **External** — depends on outside sources and must carry primary citations

## 2. Claim ledger format

Use this format for every meaningful public claim:

- **Claim**
- **Class**
- **Bound**
- **Primary evidence**
- **Limitation**
- **Safe public wording**

Example:

- **Claim:** STRONG/WEAK guard separation is a reproducible architecture pattern.
- **Class:** Proven
- **Bound:** Sentinel boundary
- **Primary evidence:** accepted internal freeze material
- **Limitation:** not yet uniformly enforced stackwide
- **Safe public wording:** “specified as doctrine and proven at the Sentinel boundary”

## 3. Proven in bounded scope

The following are safe to describe as proven in bounded scope:

- uPipe P0
- uPipe P1
- uPipe P2
- Memory Layer v1 Phase 1 authority boundary
- bounded packet lane with local signing
- uSpine as the one worked CANON example
- STRONG/WEAK guard separation at the Sentinel boundary

## 4. Partially enforced

The following are better described as partially enforced:

- receipt-driven governance across selected lanes
- contradiction and stale handling in selected surfaces
- projection-not-truth discipline in parts of the system
- review-path structures that exist but are not yet fully unified
- bounded human review integration outside the one worked CANON path

## 5. Specified, contradictory, or open

The following should remain explicitly open:

- uPipe P3 artifact gating
- generalized dispatch loop closure
- receipt-type reconciliation
- uDoc/uKnow/memory authority boundary
- final public role of uBus
- generalized sandboxing / isolation
- Bayesian operational ownership

## 6. What is intentionally out of scope

The paper must stay out of these claims:

- finished platform claims
- generalized CANON throughput
- universal artifact gating
- universal receipt unification
- complete sandboxing
- settled internal doctrine for every subsystem boundary
- volatile counts, stale snapshots, and single-source shipping claims

## 7. Truth discipline notes

### 7.1 Memory claim
Safe: the authority boundary of Memory Layer v1 Phase 1  
Unsafe: “the memory layer is complete” or “the full memory layer is proven”

### 7.2 CANON claim
Safe: human-only CANON is stable doctrine; uSpine is one worked example  
Unsafe: generalized CANON throughput claim

### 7.3 Receipt claim
Safe: receipt schema/emission exists in bounded lanes  
Unsafe: “receipts already mechanically enforce acceptance everywhere”

### 7.4 Constitutional constraints
Safe: a recurring family of constitutional constraints  
Unsafe: a frozen numbered doctrine unless one canonical primary source is explicitly cited

## 8. Open problems worth collaborator attention

The highest-value open technical problems are:

1. receipt grammar unification
2. artifact gating as a general runtime capability
3. dispatch loop closure
4. truth-boundary resolution
5. mechanically enforced sandboxing

7. advisory Bayesian integration without sovereignty leakage

## 9. Reviewer instruction

When reviewing the paper, downgrade any sentence that:
- sounds like bounded proof has become generalized proof
- turns doctrine into implementation
- turns one worked example into repeated throughput
- hides unresolved contradictions
- implies a finished stack
