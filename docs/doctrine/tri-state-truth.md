# Tri-State Truth

Tri-State Truth is the honesty protocol used to describe system reality without overclaiming.

Every meaningful claim should be labeled as one of:

## Specified

Defined in doctrine, architecture, schema, or blueprint, but not yet operating as a proven path.

## Partially Enforced

Mechanism exists and works in bounded paths, but is not yet generalized, mechanically closed, or uniformly enforced.

## Proven

Multi-source and bounded by accepted evidence. Proven means the claim survived scrutiny inside its stated bound. It does not mean “finished everywhere.”

## Why this matters

AI systems fail when design intent is narrated as implementation fact.

Tri-State Truth prevents:
- marketing upgrades
- silent generalization
- bounded proof being mistaken for platform completion
- unresolved contradictions being hidden under cleaner prose

## Rule

If a claim is bounded, say bounded.
If a claim is open, say open.
If a claim is doctrine, do not narrate it as implemented.
